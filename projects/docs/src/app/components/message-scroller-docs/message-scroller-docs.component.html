The problem

A plain overflow-y: auto div gets a streaming transcript wrong in one of two ways, and you have to pick which:

Pin to the bottom on every change

The reader scrolls up to re-read a paragraph, a token arrives, and the view yanks them back down mid-sentence. They fight the scrollbar for the rest of the reply.

Never follow

The reply streams in below the fold and the reader watches a static screen, with no signal that anything is happening until they scroll down and find it finished.

The governing rule here is never move the reader against their intent. Every automatic scroll is gated on the reader already being at the live edge. The moment they scroll up, following switches off and stays off until they come back down or press the jump button.

It also handles the two cases that are easy to forget: opening a saved thread at the newest turn rather than at the absolute end of a wall of text, and holding position when older history loads in above the viewport instead of letting the transcript lurch downward.

Installation

Add it with the CLI (you own the source), or import the parts if you use the package.

Anatomy

Note that tolleMessageScrollerItem is a directive, not a wrapper component — you put it on your own element, so it adds no extra box to your layout.

The scroller frame is flex-1 with min-h-0, so it fills a parent that has a definite height. Give that parent the height — the scroller does not set one for you.

Examples

Basic

Scroll up and the jump button appears; scroll back to within threshold px of the bottom and it disappears again, because following has resumed.

Start position and anchors

Where should a reopened thread land? 'bottom' drops the reader at the last line of the last answer — the end of something they never read the start of. 'last-anchor' instead opens at the newest turn, so a long reply starts from its first line, with scrollPreviousItemPeek px of the previous turn left visible so the reader can see what is being answered.

Turns are marked by you: set [scrollAnchor]="true" on the row that begins one. With no anchors marked, 'last-anchor' quietly falls back to 'bottom'.

The start position applies once, on mount — switch it below and the scroller is re-created so you can see the difference.

Streaming

This is the case the component exists for. Press Stream a reply and let it run: while you are at the bottom, the view follows each word. Now scroll up a line or two — following stops instantly, the jump button appears, and the text keeps arriving without moving anything under you. Press the button (or scroll back down) and it picks up again.

Streamed tokens use an instant scroll rather than a smooth one, so a hundred chunks do not queue a hundred animations.
